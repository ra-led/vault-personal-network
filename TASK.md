Техническое задание

MVP VPN-сервиса на WireGuard с Telegram-ботом

1. Цель проекта

Разработать MVP сервиса продажи доступа к VPN через Telegram-бота без собственного мобильного приложения.

Пользовательский сценарий:
	1.	Пользователь заходит в Telegram-бот
	2.	Получает инструкцию и ссылки на WireGuard-клиент
	3.	Пополняет баланс через Telegram Payments
	4.	Добавляет одно или несколько устройств
	5.	Для каждого устройства получает:
	•	файл конфигурации .conf
	•	QR-код
	6.	Импортирует конфиг в официальный клиент WireGuard
	7.	Использует VPN
	8.	Оплата списывается по дням и по устройствам

⸻

2. Основные бизнес-правила

2.1. Модель оплаты
	•	Стоимость: 2 рубля в день за 1 устройство
	•	Баланс пользователя хранится в рублях
	•	Пополнение баланса через Telegram-бота:
	•	10 рублей
	•	50 рублей
	•	100 рублей

2.2. Списание средств
	•	Списание происходит ежедневно
	•	Формула:
	•	количество активных устройств × 2 рубля в день
	•	Если на балансе недостаточно средств:
	•	все активные устройства пользователя переводятся в состояние suspended
	•	доступ к VPN для них отключается

2.3. Устройства
	•	Один аккаунт Telegram может иметь несколько устройств
	•	Каждое устройство:
	•	имеет своё имя
	•	имеет отдельный WireGuard peer
	•	имеет отдельный конфиг
	•	учитывается отдельно в биллинге
	•	может быть заблокировано/удалено отдельно

⸻

3. Общая архитектура

Проект должен состоять из двух Docker Compose стеков.

3.1. Control Plane

Центральный сервер, отвечающий за:
	•	Telegram-бота
	•	API
	•	биллинг
	•	БД
	•	выбор VPN-ноды
	•	генерацию конфигов
	•	выдачу QR и файлов
	•	хранение пользователей, устройств, платежей, балансов
	•	мониторинг нод
	•	баны и блокировки

3.2. Edge Node

VPN-нода, отвечающая за:
	•	запуск WireGuard-сервера
	•	регистрацию в центре
	•	создание и удаление peers
	•	сбор статистики трафика
	•	отправку heartbeat в control plane
	•	исполнение команд центра

⸻

4.1. Язык и framework

6. Функциональные требования

6.1. Telegram-бот

6.1.1. Главное меню

Бот должен поддерживать следующие разделы:
	•	Инструкция
	•	Баланс
	•	Пополнить баланс
	•	Мои устройства
	•	Добавить устройство
	•	Поддержка или Помощь

6.1.2. Раздел “Инструкция”

Бот должен отправлять:
	•	краткое описание, как подключиться
	•	ссылки на приложения WireGuard:
	•	iOS
	•	Android
	•	Windows
	•	macOS
	•	краткую инструкцию:
	•	установить приложение
	•	добавить конфиг через файл или QR
	•	включить туннель

6.1.3. Раздел “Баланс”

Бот показывает:
	•	текущий баланс в рублях
	•	количество активных устройств
	•	ежедневное списание
	•	прогноз на сколько дней хватит баланса

Пример:
	•	Баланс: 50 ₽
	•	Активных устройств: 2
	•	Списание: 4 ₽/день
	•	Хватит примерно на 12 дней

6.1.4. Пополнение баланса

Бот должен предоставлять кнопки:
	•	10 ₽
	•	50 ₽
	•	100 ₽

После успешной оплаты:
	•	баланс пользователя увеличивается
	•	создаётся запись о платеже
	•	бот отправляет подтверждение

6.1.5. Добавление устройства

При добавлении устройства:
	1.	Бот спрашивает имя устройства
	2.	Проверяет, что у пользователя есть положительный баланс
	3.	Запрашивает создание peer через control plane
	4.	Получает:
	•	.conf
	•	QR-код
	5.	Отправляет пользователю:
	•	файл .conf
	•	QR-картинку
	•	краткую инструкцию по импорту

6.1.6. Мои устройства

Для каждого устройства отображать:
	•	имя
	•	статус
	•	дата создания
	•	нода/страна
	•	объём переданного трафика
	•	кнопки:
	•	Получить конфиг заново
	•	Переименовать
	•	Отключить
	•	Удалить

6.1.7. Получить конфиг заново

При запросе:
	•	перевыпускается конфиг по логике безопасности

6.1.8. Удаление устройства

При удалении устройства:
	•	peer удаляется с edge-node
	•	устройство переводится в удалённое/деактивированное состояние
	•	ежедневное списание за него прекращается

7. Логика биллинга

7.1. Ежедневный расчёт

Один раз в сутки scheduler:
	•	получает список пользователей
	•	считает число активных устройств
	•	рассчитывает сумму списания:
	•	активные устройства × 2 ₽
	•	если денег хватает:
	•	уменьшает баланс
	•	если денег не хватает:
	•	переводит все активные устройства пользователя в suspended
	•	инициирует отключение peers на edge-node

7.2. Возобновление после пополнения

Если пользователь пополняет баланс:
	•	suspended устройства автоматически восстановливаются

⸻

8. Управление устройствами и VPN-peer

8.1. Каждое устройство = отдельный WireGuard peer

Для каждого устройства создаются:
	•	client private key
	•	client public key
	•	internal VPN IP
	•	конфигурационный файл

8.2. Поля WireGuard-конфига

Конфиг должен содержать:
	•	PrivateKey
	•	Address
	•	DNS
	•	PublicKey сервера
	•	Endpoint
	•	AllowedIPs
	•	PersistentKeepalive

8.3. Ограничения
	•	Один peer привязан к одному устройству
	•	Устройство должно быть связано с конкретной edge-node
	•	Устройство может быть:
	•	active
	•	suspended
	•	banned
	•	deleted

⸻

9. Edge-node и авто-регистрация

9.1. Регистрация ноды

При запуске edge-node должна:
	•	подключаться к control plane
	•	передавать:
	•	имя ноды
	•	внешний IP
	•	hostname
	•	страну/локацию
	•	максимальное количество клиентов
	•	версию агента
	•	получать внутренний node_id и токен

9.2. Heartbeat

Edge-node должна отправлять heartbeat, например раз в 30–60 секунд, содержащий:
	•	статус
	•	число активных peers
	•	tx/rx по ноде
	•	загрузку CPU
	•	свободное место на диске
	•	время последней активности

9.3. Выбор ноды

При создании нового устройства control plane должен выбирать ноду:
	•	только среди healthy
	•	с минимальной загрузкой
	•	с запасом по client slots

Для MVP достаточно самого простого алгоритма:
	•	выбрать healthy ноду с минимальным числом активных клиентов

⸻

10. Учёт трафика

10.1. Требования

Система должна учитывать трафик:
	•	по каждому устройству
	•	по каждой ноде
	•	суммарно по пользователю

10.2. Источник данных

Edge-node должна собирать статистику по peers:
	•	входящий трафик
	•	исходящий трафик

10.3. Сохранение

Control plane должен сохранять агрегированные данные в БД.

Минимум:
	•	ежедневная статистика по каждому устройству

10.4. Отображение в боте

В “Мои устройства” пользователь должен видеть:
	•	трафик по устройству

Для MVP достаточно:
	•	суммарный tx/rx за всё время
	•	либо за текущий день

⸻

11. Баны и блокировки

11.1. Уровни блокировки

Система должна поддерживать:

Блокировка устройства
	•	отключение одного peer

Блокировка пользователя
	•	отключение всех устройств пользователя
	•	запрет на создание новых устройств

Блокировка по недостатку баланса
	•	временный suspend

11.2. Причины блокировки
	•	manual admin ban
	•	abuse
	•	недостаточно средств
	•	превышение лимитов, если будут введены позже

11.3. Исполнение блокировки

При блокировке:
	•	control plane отправляет команду edge-node
	•	edge-node удаляет peer или помечает его как неактивный

13. Логи и очистка

13.1. Общие требования к логам

Логировать:
	•	регистрацию пользователей
	•	платежи
	•	создание/удаление устройств
	•	создание/удаление peers
	•	heartbeat нод
	•	ошибки API
	•	ошибки edge-agent

Не логировать:
	•	полный пользовательский трафик
	•	DNS-запросы
	•	содержимое конфигов целиком в plaintext
	•	client private keys в логах

13.2. Ротация логов

Необходимо настроить ротацию логов Docker, чтобы они не заполнили диск.

Минимальные требования:
	•	ограничение размера лог-файла
	•	ограничение числа ротаций

Пример целевого поведения:
	•	10 MB на файл
	•	3 архивных файла

13.3. Очистка

Нужен job на очистку:
	•	просроченных временных файлов
	•	старых QR-кодов
	•	неиспользуемых артефактов генерации конфигов

⸻

14. Безопасность

14.1. Хранение ключей
	•	client private keys не должны храниться в открытом виде в логах
	•	в БД должны храниться либо в зашифрованном виде, либо должен использоваться безопасный способ временной генерации

Для MVP допустимо:
	•	хранение private key в БД в зашифрованном виде

14.2. API-доступ edge-node
	•	edge-node должна аутентифицироваться в control plane по токену
	•	все запросы между edge-node и control plane должны идти по HTTPS

14.3. Telegram webhook/API
	•	бот должен быть защищён от повторных запросов и ошибок идемпотентности
	•	платёжные webhook должны обрабатываться идемпотентно

14.4. Доступ к конфигам

Поскольку бот присылает файл и QR напрямую в Telegram, отдельная публичная ссылка на конфиг в MVP не обязательна.

16. API control plane

16.1. Внутреннее API для edge-node

Регистрация ноды

POST /internal/nodes/register

Heartbeat

POST /internal/nodes/heartbeat

Отправка usage

POST /internal/nodes/usage

Создание peer

POST /internal/nodes/{node_id}/peers

Удаление peer

DELETE /internal/nodes/{node_id}/peers/{device_id}

Suspend peer

POST /internal/nodes/{node_id}/peers/{device_id}/suspend

Resume peer

POST /internal/nodes/{node_id}/peers/{device_id}/resume

16.2. Внешнее API

Может использоваться ботом и админкой:
	•	получить профиль пользователя

	•	получить статус нод

19. Cron/scheduler задачи

Необходим scheduler, выполняющий периодические задачи:

Раз в сутки
	•	списание баланса за активные устройства
	•	suspend устройств при нехватке средств

Раз в 1–5 минут
	•	проверка состояния edge-node
	•	обработка heartbeat timeout
	•	синхронизация usage
	•	очистка временных файлов

Раз в 1 час
	•	перепроверка suspended устройств
	•	обновление агрегированной статистики
